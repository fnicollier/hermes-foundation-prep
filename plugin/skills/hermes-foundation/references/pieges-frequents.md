# Pièges fréquents observés dans la banque officielle (TÜV SÜD)

Liste des pièges récurrents extraits des 40 questions officielles. À utiliser comme inspiration pour les distracteurs et comme révision de fin de parcours.

## 1. Comité spécialisé = organisation permanente, PAS organisation de projet

**Source : Q26.**

Bien que le Comité spécialisé soit attribué au niveau hiérarchique du **pilotage**, il n'est **pas** un rôle de l'organisation de projet — c'est un rôle de l'**organisation permanente** qui intervient en consultation.

Distracteurs typiques : « Conduite », « Exécution », « Pilotage » (ces 3 sont des niveaux hiérarchiques, mais le piège est qu'aucun n'est la bonne réponse — la bonne réponse est *« Aucun. Le comité spécialisé est un rôle de l'organisation permanente. »*).

## 2. Organisation de projet effective avec le mandat d'initialisation

**Source : Q29.**

L'organisation de projet **devient effective avec le mandat d'initialisation du projet** — pas plus tard, pas plus tôt.

Distracteurs typiques : « libération de la phase de conception », « décision relative à la suite du projet », « libération de l'exécution ».

## 3. Phases en mode classique = 5, pas 4 ni 6

**Source : Q3.**

Phases classiques : **Initialisation, Conception, Réalisation, Déploiement, Clôture** (5 phases).

Pièges courants :
- Inclure « analyse préliminaire » (n'existe pas en HERMES).
- Inclure « mise en œuvre » (c'est une phase de l'approche **agile**, pas classique).
- Oublier la clôture.

## 4. Détail suffisant pour planifier = phase de Conception

**Source : Q4.**

En classique, la phase où les résultats sont *suffisamment détaillés pour que les participants puissent planifier, présenter et réaliser la solution sur une base fiable* = **Conception**.

(Pas Initialisation, pas Réalisation.)

## 5. Listes de contrôle soutiennent la vérification des jalons

**Source : Q5.**

Les **listes de contrôle** soutiennent la vérification de l'atteinte des jalons.

Distracteurs : Mandat de travail, Méthode Delphi, Structure de découpage du projet (rien à voir).

## 6. Rapport de phase = base pour libérer la phase suivante

**Source : Q6.**

Le **rapport de phase** sert de base pour la décision de libérer ou non la phase suivante.

Faux distracteurs :
- Le plan de gestion = base formelle pour la libération de l'**exécution**, pas de la phase suivante en général.
- Les expériences acquises ne documentent pas les décisions de libération.

## 7. Annulation d'un projet en initialisation = pas de faisabilité

**Source : Q7.**

La raison qui peut justifier l'annulation d'un projet en phase d'initialisation est **pas de faisabilité** (constat dans le rapport d'évaluation).

## 8. Modèle de phases = base pour compréhension commune du déroulement

**Source : Q8.**

Le but du modèle de phases : il **repose sur le cycle de vie du projet et crée la condition préalable à la compréhension commune du déroulement du projet**.

Pas : « attribuer les tâches », « régler la responsabilité ».

## 9. Contrôle de réussite du projet = APRÈS la clôture du projet

**Source : Q9.**

Le contrôle de la réussite du projet est exécuté **quelque temps après la clôture du projet** (mesure de l'atteinte des bénéfices business).

Pas : avant chaque jalon, avant chaque clôture de module, avant la clôture de la phase de déploiement.

## 10. Module Organisation = définit l'organisation de projet ET l'adaptation

**Source : Q15.**

Le module **Organisation** définit notamment l'**organisation de projet** elle-même (et adapte aussi l'organisation permanente).

Pièges :
- *Organisation du déploiement* est un autre module distinct.
- *Organisation d'exploitation* relève de l'exploitation IT.

## 11. Rôles indispensables = Mandant + Chef de projet + Représentant des utilisateurs

**Source : Q21.**

Trinité incontournable : **Mandant / Chef de projet / Représentant des utilisateurs** (les 3 dans le groupe Utilisateurs).

Pièges :
- Comité de pilotage / Comité spécialisé : pas indispensables.
- Business analyst, Spécialiste : pas obligatoires.

## 12. Pilotage = Mandant + Comité de pilotage + Gestionnaire QR ? NON

**Source : Q22.**

Niveau hiérarchique du **pilotage** : Mandant + Comité de pilotage + **Comité spécialisé** (pas le Gestionnaire qualité & risques, qui est en conduite, sous le Chef de projet).

## 13. Chef de projet = compétence sur les ressources LIBÉRÉES

**Source : Q25.**

Le chef de projet a la compétence concernant l'utilisation des **ressources libérées** au projet — pas toutes les ressources humaines de l'organisation.

## 14. Gestionnaire qualité & risques = subordonné au Chef de projet

**Source : Q31.**

Le Gestionnaire de la qualité et des risques est subordonné au **Chef de projet**, pas au Mandant ni au Comité de pilotage.

## 15. Représentant des utilisateurs = interface Utilisateurs ↔ Producteurs/Exploitants

**Source : Q24.**

C'est le **Représentant des utilisateurs** qui sert d'interface entre les groupes de partenaires Utilisateurs et Producteurs/Exploitants.

Pas le Business analyst (qui traduit en exigences techniques côté producteur).

## 16. 3 groupes de partenaires = Utilisateur / Producteur / Exploitant

**Source : Q27.**

Les 3 groupes de partenaires sont : **Utilisateur / Producteur / Exploitant**.

Pièges :
- *Bénéficiaire de prestations* (mauvais terme).
- *Prestataire / Fournisseur* (mauvais découpage).

## 17. Décider de la suite du projet = Comité de pilotage

**Source : Q23.**

La tâche **« Décider de la suite du projet »** est de la responsabilité du **Comité de pilotage**.

(Le Mandant décide certains jalons, mais la suite du projet relève collégialement du Comité.)

## 18. Description d'un rôle = Description / Responsabilité / Compétences / Qualifications

**Source : Q34.**

La structure de description d'un rôle dans HERMES est : **Description, responsabilité, compétences, qualifications** (et relations selon édition).

Pièges :
- *Domaine d'activité* à la place de compétences.
- Oublier les qualifications.

## 19. Niveau hiérarchique de l'exécution = élaborer la solution + AQ

**Source : Q37.**

Au niveau **exécution** : élaborer la solution **ET** mettre en œuvre les mesures d'assurance qualité (les deux).

Pièges :
- Confondre avec la conduite (qui *garantit* l'AQ mais ne l'exécute pas).
- Confondre avec le pilotage (qui pilote, ne réalise pas).

## 20. Représentant des utilisateurs = représenter les utilisateurs et leurs intérêts

**Source : Q38, Q40.**

La description du Représentant des utilisateurs : il **représente les utilisateurs et leurs intérêts dans le contexte du projet**, détermine les exigences sur la base des processus métier, en définit les priorités, et les transforme en exigences pour le système.

Pas :
- *Soutenir le chef de projet sur les questions admin* (= Assistance de projet).
- *Garantir l'entretien et la suite du développement* (= organisation permanente après projet).
- *Élaborer les exigences organisationnelles* (= module Organisation, pas le rôle Représentant).

## Récapitulatif visuel — pièges par chapitre

|------------------|-----------------|

| Phases | Confondre « mise en œuvre » (agile) avec « réalisation » (classique) |
| Jalons | Oublier que les jalons sont des résultats de type *état* |
| Modules | Confondre Organisation et Organisation du déploiement |
| Modules | Oublier que Bases du projet est obligatoire en agile aussi |
| Rôles | Comité spécialisé attribué à pilotage, mais c'est un rôle de l'org. permanente |
| Rôles | Compter le Gestionnaire QR au pilotage (faux : conduite) |
| Rôles | Compter le Chef de projet au pilotage (faux : conduite) |
| Org. projet | Devient effective… pas au mandat d'exécution mais au mandat d'initialisation |
| Tâches | Décider de la suite du projet : Mandant (faux : Comité de pilotage) |
| Groupes partenaires | Utilisateur/Prestataire/Fournisseur (faux ; c'est Utilisateur/Producteur/Exploitant) |
